Functions in JavaScript and Ruby compared!

Subject: Functions in Ruby and Java script

23/03/2015

What are functions?

Once you get your head around what a function does, then learning the various syntactic idiosyncrasies of Ruby and JavaScript are the only important things in understanding functions for both of these languages, because both languages have their own ways of expressing the same idea of what a function should do and fortunately enough, there is a lot of overlap between how functions operate in both Ruby and JavaScript.

So what is a function? Simply put a function takes an input, performs some alteration to the input, and then produces an output. I think it’s easy to conceptualize a function with the help of the Mr. Men character series. Mr.Men characters are a good analogy to functions since they too operate on a set of rules which guides their behavior and output in the world.

Imagine if you asked Mr Loud to say the word: spaghetti. Being Mr Loud how operates by a set of rules which makes all input he receives louder, he will probably output: "SPAGHETTI" in all uppercase. Imagine if you asked Mr Nonsense to repeat the words "Delicious Fudge", being Mr Non Sense he might return your input in gibberish form; where each letter you told him as input has been randomly shuffled to produce an output as follows: "Iuege Ddlousfci!" "

Think of the Mr. Men characters as functions; they have a set of rules; Mr Shout will always shout; and return your input as a string in uppercase. Mr Nonsense will always shuffle your characters.

A function like square will square a number you feed in. And more complicated functions can be two, three or four steps long, performing calculations to the input which you feed them and producing a desired output.

Here is an example of a function being defined in Ruby:

def times_three(n) {
return n*3
};

In the two languages of JavaScript and Ruby functions are written in different ways, however they accomplish the same goals, manipulating input with a set of rules to create a different output. In Ruby a function is defined with the key word def. The (n) after the function name times_three takes an argument which can be any number which we choose to feed in once we call the function. The block of code after the a (n) in the curly braces is what happens to input to produce output.

Calling "times_three(4)" this will produce the output 12. We can now calculate the result of any number multiplied by 3, by feeding in the number as an argument in our new function. This helps keep our code DRY as we will not repeat ourselves.

In both languages once functions are defined they can be invoked or called, later in the code, by typing out the function and feeding in optional arguments, depending on if the function was defined with parameters.

Inside the functions in JavaScript and Ruby the arguments are used as local variables. All variables defined within a function have a local scope in both languages. It is possible to reference and use a variable which has been defined outside of a function however you cannot references a variable which has been defined as inside a function. Think of variables defined within a function as being within a yellow submarine. They can peek out above them with a periscope at other variables around them, but variables outside the submarine can’t peek in at them.

When both languages reach the word return in their function, they will stop executing. In Ruby if you don’t specify a return value, the last line of the code is evaluated and returned as if the word return was present. The return values is returned back to the “caller” which is usually the method.

A big difference in functions in these two languages is that Ruby will return an argument error if there are not the exact number of arguments typed into the function as the parameters which we set up when creating the function. JavaScript is more relaxed about this though, it will not return a fatal error, instead it will return the word undefined. Although this sounds useful since we are not receiving an error it can actually be a problem when writing code because it can be difficult to spot an argument issue in our code because everything appears to be working just fine!
